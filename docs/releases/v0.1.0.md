# danielsmith.io v0.1.0 launch plan

This is the concrete v0.1.0 release checklist for promoting `danielsmith.io` from prototype
portfolio content to the production service at `https://danielsmith.io`. It is intentionally a
planning document only: do not implement the launch page, metrics stack, dashboards, alerts,
deployment changes, Git tags, package versions, or chart versions from this checklist.

## Current evidence and scope boundaries

- [ ] Confirm the production root serves the immutable image built from this repository, not a
      legacy placeholder or unrelated host. The 2026-06-20 spot check reached Cloudflare/envoy and
      returned `text/html` for `/`, `/livez`, `/healthz`, and `/resume.pdf`; that does not prove the
      current production host is serving this repository's nginx runtime because this repository's
      nginx config returns JSON for `/livez` and `/healthz`.
- [ ] Confirm `/resume.pdf` is served as the stable resume PDF path before making it a paging
      production probe. The root resume contract is planned in the resume workflow notes, but current
      production spot checks returned `text/html`, not a PDF response.
- [ ] Treat `/runtime/github-metrics.json` and GitHub stars/forks as portfolio metadata only. They
      may remain on a dashboard in a clearly labeled product/content panel, but they are not uptime,
      latency, readiness, saturation, or release-health signals.
- [ ] Treat browser-local performance and failover events as future design inputs only. v0.1.0 must
      not require or add a hidden public analytics collector.

## Launch surface requirements

- [ ] `https://danielsmith.io/` production serves the application from this repository and the
      promoted immutable image, rather than a legacy placeholder page.
- [ ] The default landing or text surface is simple, accessible, and exposes clear links to:
  - [ ] Resume at `/resume.pdf`.
  - [ ] GitHub.
  - [ ] LinkedIn.
  - [ ] Email.
  - [ ] DSPACE.
  - [ ] token.place.
- [ ] The immersive experience provides a usable, keyboard-reachable route to the same links.
- [ ] The text fallback provides a usable route to the same links without requiring WebGL.
- [ ] The resume link uses the planned stable `/resume.pdf` path rather than a dated runtime URL.
- [ ] Keyboard navigation remains a release gate: visitors can reach launch links, mode controls,
      help controls, POIs, and fallback links without a pointer.
- [ ] Visible focus remains a release gate for launch links and interactive HUD/fallback controls.
- [ ] Screen-reader labeling remains a release gate for launch links, mode state, fallback reason,
      help controls, and POI/fallback content.
- [ ] Reduced-motion behavior remains a release gate for animated launch content, POI highlights,
      fallback transitions, and accessibility presets.
- [ ] Text fallback remains a release gate and must work when WebGL is unavailable, low-memory
      heuristics fail the device, social/link-preview crawlers request the page, or `?mode=text` is
      used.
- [ ] Staging is validated before the exact same immutable `main-<shortsha>` image is promoted to
      production.

## Required baseline observability slice

v0.1.0 observability should stay small and appropriate for a static nginx application. Prefer
Sugarkube-owned Prometheus, blackbox exporter, kube-state-metrics, node/kubelet/cAdvisor metrics,
and dashboard/alert rules over adding app code.

### Public blackbox probes

- [ ] Probe `https://staging.danielsmith.io/` and `https://danielsmith.io/`.
- [ ] Probe `https://staging.danielsmith.io/livez` and `https://danielsmith.io/livez`.
- [ ] Probe `https://staging.danielsmith.io/healthz` and `https://danielsmith.io/healthz`.
- [ ] Probe `https://staging.danielsmith.io/resume.pdf` and
      `https://danielsmith.io/resume.pdf` after the stable PDF route serves or redirects to the actual
      PDF.

### Metrics and signals

- [ ] Public success rate from blackbox `probe_success`, grouped by environment and path.
- [ ] Public latency from blackbox `probe_duration_seconds`, grouped by environment and path.
- [ ] TLS certificate expiry from blackbox TLS metrics.
- [ ] Kubernetes deployment readiness from kube-state-metrics.
- [ ] Pod restart count from Kubernetes/container metrics.
- [ ] CPU saturation from container/node metrics against the app's requested/limited CPU.
- [ ] Memory saturation from container/node metrics against the app's requested/limited memory.
- [ ] Current immutable image/release identity from Kubernetes deployment/pod labels, image tag, or
      an equivalent low-cardinality release label.
- [ ] Prometheus scrape health for any app-owned or platform-owned scrape target.
- [ ] Blackbox target health for each staging and production probe.

### Provisional thresholds

Use these as starting points only. Staging data must establish real baselines before final
production thresholds are treated as stable.

| Signal               | Provisional threshold                           | Window              | Initial action                                                                |
| -------------------- | ----------------------------------------------- | ------------------- | ----------------------------------------------------------------------------- |
| Public root success  | `probe_success == 0`                            | 3m prod, 5m staging | Check Cloudflare/DNS, ingress, pod readiness, then rollback if image-related. |
| Health success       | `probe_success == 0` for `/livez` or `/healthz` | 3m prod, 5m staging | Check deployment rollout, nginx config, and probes.                           |
| Resume PDF success   | `probe_success == 0` for `/resume.pdf`          | 5m prod             | Check static asset/alias generation and rollback if release-related.          |
| Public latency       | p95 > 2s for static routes                      | 15m                 | Compare with staging baseline, Cloudflare path, and node saturation.          |
| TLS expiry           | <14d warning, <3d critical                      | 1h                  | Fix cert-manager/Cloudflare; do not rollback app by default.                  |
| Pod restarts         | >3 restarts                                     | 15m                 | Inspect events/logs; rollback only for release regressions.                   |
| Deployment readiness | available replicas below desired                | 10m                 | Inspect rollout status and previous ReplicaSet; rollback if needed.           |
| Scrape health        | `up == 0` for expected scrape target            | 10m                 | Fix ServiceMonitor/static scrape; do not page if public app is healthy.       |
| Blackbox health      | missing/failed target                           | 5m                  | Fix blackbox config or endpoint route, depending on cause.                    |

## Dashboard requirement

- [ ] Create a `danielsmith.io` dashboard in Sugarkube/Grafana after real metrics exist.
- [ ] Include public endpoint status by path and environment.
- [ ] Include latency history by path and environment.
- [ ] Include TLS certificate expiry for staging and production hosts.
- [ ] Include pod readiness and restart history.
- [ ] Include CPU and memory usage/saturation.
- [ ] Include staging versus production comparisons through an `environment` variable or separate
      rows.
- [ ] Include the deployed immutable tag or equivalent release/image identity.
- [ ] If GitHub portfolio metadata is retained, place it in a separate panel clearly labeled
      `Portfolio metadata (not operational health)`.

## Alert requirement

Enable only alerts with a clear operator action and runbook path.

- [ ] Production root unavailable.
- [ ] Health endpoint unavailable for `/livez` or `/healthz`.
- [ ] Resume PDF unavailable after `/resume.pdf` is implemented as the stable PDF route.
- [ ] TLS expiry approaching.
- [ ] Repeated pod restart.
- [ ] Deployment unavailable after rollout.

## Privacy boundaries

- [ ] Existing browser performance and failover events may inform a future privacy-reviewed
      telemetry design, but they are not v0.1.0 production-promotion requirements.
- [ ] v0.1.0 must not add a public analytics collector as a hidden requirement.
- [ ] No IP address, browser fingerprint, precise device identity, navigation history, or visitor
      identifier may enter Prometheus labels.
- [ ] Client-side performance collection must remain local by default unless a later design defines
      aggregation, consent, retention, and deletion behavior.
- [ ] Prometheus labels must stay bounded: app, environment, cluster, namespace, release, route
      group/path from the approved probe list, and status/outcome categories are acceptable.
- [ ] Do not label metrics, dashboard rows, or alerts with raw URLs, arbitrary query strings, user
      agents, request IDs, exception text, or visitor-specific values.

## Explicit post-v0.1.0 candidates

These are deliberately out of scope for the v0.1.0 launch gate:

- [ ] Privacy-safe aggregate browser performance telemetry.
- [ ] nginx request metrics or an exporter.
- [ ] Richer WebGL/failover dashboards.
- [ ] Loki or centralized logs.
- [ ] Distributed tracing.

## Promotion evidence checklist

Record command output, screenshots, dashboard links, and immutable tags in the release record. Do
not check off items until repository, cluster, or dashboard evidence proves completion.

### Build artifact and staging

- [ ] Choose a successful `main` image workflow run and record the immutable tag:
      `ghcr.io/futuroptimist/danielsmith.io:main-REPLACE_SHORTSHA`.
- [ ] Deploy staging from the Sugarkube checkout with the immutable tag:

  ```bash
  just danielsmith-oci-deploy env=staging tag=main-REPLACE_SHORTSHA
  ```

- [ ] Confirm staging rollout and image identity:

  ```bash
  kubectl -n danielsmith rollout status deploy/danielsmith --timeout=180s
  kubectl -n danielsmith get deploy danielsmith -o wide
  kubectl -n danielsmith get pods -l app.kubernetes.io/name=danielsmith -o wide
  ```

### Staging endpoint checks

- [ ] Root check succeeds:

  ```bash
  curl -fsS https://staging.danielsmith.io/
  ```

- [ ] Liveness check succeeds:

  ```bash
  curl -fsS https://staging.danielsmith.io/livez
  ```

- [ ] Health check succeeds:

  ```bash
  curl -fsS https://staging.danielsmith.io/healthz
  ```

- [ ] Stable resume PDF check succeeds and returns a PDF response:

  ```bash
  curl -fsSI https://staging.danielsmith.io/resume.pdf
  ```

### Staging observability checks

- [ ] Prometheus discovers the expected scrape targets or explicitly records that only blackbox and
      Kubernetes metrics are in scope for v0.1.0.
- [ ] Blackbox targets for `/`, `/livez`, `/healthz`, and `/resume.pdf` are discovered and healthy
      in staging.
- [ ] The `danielsmith.io` dashboard shows staging endpoint status, latency, TLS expiry, pod
      readiness/restarts, resource usage, and immutable image identity.
- [ ] One controlled alert test is performed in staging and evidence is captured. Prefer a safe
      blackbox target or alert-rule test; do not break production to prove alerting.

### Accessibility and fallback smoke evidence

- [ ] Keyboard-only smoke evidence covers the launch links and key HUD/fallback controls.
- [ ] Visible-focus evidence covers the launch links and controls.
- [ ] Screen-reader labeling evidence covers the launch links, mode/fallback state, and primary
      controls.
- [ ] Reduced-motion evidence covers the immersive route and text fallback.
- [ ] Text fallback evidence covers `?mode=text` and at least one automatic fallback trigger or
      crawler/headless route.

  ```bash
  just danielsmith-oci-promote-prod tag=main-REPLACE_PRIOR_SHORTSHA
  ```

- [ ] Verify rollback rollout status and image identity:

  ```bash
  kubectl -n danielsmith rollout status deploy/danielsmith --timeout=180s
  kubectl -n danielsmith get deploy danielsmith -o wide
  ```

- [ ] Repeat post-promotion root, health, resume, blackbox, and dashboard verification after
      rollback.
